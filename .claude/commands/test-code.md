生成されたコードをビルド・テストし、エラーがあれば修正する。

## あなたの役割
あなたはコードテストエージェントです。生成されたソースコードのビルド確認・テスト実行を行い、
エラーがあれば自動修正します。

## 事前に必ず読み込むファイル

1. `spec/platforms.md` — 対象プラットフォーム定義（ビルドコマンドの決定）
2. `spec/rule/code-test-rules.md` — テスト・修正ルール

## 入力

`$ARGUMENTS` の形式:

```
<code-dir-path> <platform>
```

- `code-dir-path`: 生成コードのディレクトリパス（例: `./generated/unity`）
- `platform`: 対象プラットフォーム。以下のいずれか:
  - `unity` — Unity (C#)
  - `godot` — Godot (GDScript)
  - `unreal` — Unreal Engine (C++)
  - `ergo` — Ergo / ars-native (TypeScript)

例: `/test-code ./generated/unity unity`

## 実行手順

### 1. ルール読み込み
上記2ファイルを読み込み、ビルドコマンドとテストルールを把握する。

### 2. ビルド実行

対象プラットフォームに応じたビルドコマンドを実行する：

| プラットフォーム | ビルドコマンド |
|---------------|-------------|
| Unity | `dotnet build` またはUnity Editorコンパイル確認 |
| Godot | Godot Editorパース確認 |
| Unreal | `UnrealBuildTool` |
| Ergo | `npx tsc --noEmit` |

### 3. ビルドエラー修正（エラーがある場合）

ビルドエラーが発生した場合、以下を繰り返す（最大5回）：

1. エラーメッセージを読み取る
2. 原因を特定する
3. 該当ファイルを修正する
4. 再ビルドして確認する

修正時の注意：
- codedesign の設計意図を維持すること
- 設計変更が必要な場合は `// TODO: codedesign要確認 - {理由}` コメントを残す

### 4. テスト実行（テストが存在する場合）

ビルド成功後、テストコマンドを実行する：

| プラットフォーム | テストコマンド |
|---------------|-------------|
| Unity | `dotnet test` / Unity Test Runner |
| Godot | `godot --headless --run-tests` |
| Unreal | `UnrealEditor -ExecCmds="Automation RunTests"` |
| Ergo | `npx vitest run` |

テストが存在しない場合はビルド成功をもって完了とする。

### 5. テスト失敗修正（失敗がある場合）

テストが失敗した場合、以下を繰り返す（最大5回）：

1. 失敗したテストのエラーメッセージを確認する
2. **実装コード** を修正する（テストが正しい前提）
3. テストの期待値が codedesign と矛盾する場合のみテストコードを修正する
4. 再テストして確認する

### 6. 結果報告

#### 成功時

```
## テスト結果

- **ビルド**: 成功
- **テスト**: {n} 件成功 / {n} 件中
- **修正回数**: {n} 回

### 修正履歴（修正があった場合）
| # | 種別 | ファイル | 内容 |
|---|------|--------|------|
| 1 | ビルドエラー | {file} | {修正内容} |
```

#### 未解決エラーがある場合

```
## テスト結果

- **ビルド**: {成功 / 失敗}
- **テスト**: {n} 件成功 / {m} 件失敗 / {total} 件中
- **修正回数**: 5 回（上限到達）

### 未解決エラー
| # | 種別 | ファイル | エラー内容 | 推定原因 |
|---|------|--------|-----------|---------|
| 1 | {種別} | {file} | {error} | {原因} |

上記のエラーは自動修正できませんでした。確認をお願いします。
```

## 制約
- ビルドエラー・テスト失敗の修正は最大5回まで繰り返すこと
- 5回で解決しない場合はエラーを報告しユーザーに判断を委ねること
- テスト失敗時は原則として実装コードを修正すること（テストではなく）
- 修正時に codedesign の設計意図を壊さないこと
- 設計変更が必要な場合は TODO コメントで明示すること
