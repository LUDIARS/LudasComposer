# codedesign レビュールール

生成された codedesign ファイルをレビューし、設計品質を検証するためのルール。

> **位置づけ**: コード生成フローの第2フェーズ。第1フェーズで生成した codedesign を検証・修正し、
> 第3フェーズ（コード生成）への入力品質を担保する。

## レビューフロー

```
[codedesign/ のMDファイル群]
    ↓ 読み込み
[REVIEW.md 読み込み（プロジェクト内に存在する場合）]
    ↓ レビュー基準の決定
[全ファイルを基準に照らしてチェック]
    ↓ 結果出力
[問題一覧 + 修正案を報告、または自動修正]
```

## レビュー基準の決定

### REVIEW.md が存在する場合

プロジェクト内の `spec/REVIEW.md` を読み込み、その中に定義されたレビュー基準を **全て** 適用する。
加えて、本ルールの「共通チェック項目」も併せてチェックする。

### REVIEW.md が存在しない場合

**単一責任の原則（SRP）チェックのみ** を行う：

- 各アクターが1つの明確な責務を持っているか
- 各コンポーネントが1つの関心事に集中しているか
- 1つのアクターに無関係な責務が混在していないか

## 共通チェック項目

REVIEW.md の有無に関わらず、以下は常にチェックする。

### C1: 構造整合性

| チェック | 内容 |
|---------|------|
| ファイル存在 | 全アクターに対応する設計ファイルが `codedesign/` 内に存在するか |
| 命名規則 | ファイル名・ディレクトリ名が kebab-case に従っているか |
| テンプレート準拠 | `codedesign-generation-rules.md` のテンプレートの全セクションが存在するか |
| シーン定義 | 各シーンディレクトリに `_scene.md` が存在するか |

### C2: メッセージ整合性

| チェック | 内容 |
|---------|------|
| 双方向一致 | アクターAの「送信メッセージ」がアクターBの「受信メッセージ」に記載されているか |
| インタフェース出力 | `messageType: interface` のメッセージが `interfaces/` にも出力されているか |
| アクション紐付け | メッセージに紐づくアクションがアクション定義に存在するか |
| 送受信先存在 | メッセージの送信先・受信元として記載されたアクターが実在するか |

### C3: コンポーネント整合性

| チェック | 内容 |
|---------|------|
| 詳細展開 | コンポーネントのIDだけでなく、変数・タスク・依存関係が展開されているか |
| 依存解決 | コンポーネントの依存先コンポーネントがプロジェクト内に存在するか |
| カテゴリ整合 | コンポーネントのカテゴリ（UI/Logic/System/GameObject）が適切か |

### C4: プラットフォーム整合性

| チェック | 内容 |
|---------|------|
| 言語一致 | メタ情報の「実装言語」が対象プラットフォームに対応しているか |
| 実装ノート | プラットフォーム別実装ノートが記載されているか |
| レンダリング | Display を持つアクターにプラットフォーム別レンダリング指示があるか |

## SRP（単一責任の原則）チェック詳細

REVIEW.md がない場合に適用する唯一のチェック。

### S1: アクターレベルの SRP

各アクターについて以下を確認する：

- **概要（overview）** が1つの責務を表現しているか
  - NG例: 「プレイヤーの移動と敵のAIを管理する」（2つの責務が混在）
  - OK例: 「プレイヤーキャラクターの移動と入力処理を管理する」
- **達成目標（goals）** が同一ドメインに収まっているか
- **コンポーネント構成** に無関係なドメインのモジュールが混在していないか

### S2: コンポーネントレベルの SRP

各コンポーネントについて以下を確認する：

- **タスク群** が同一の関心事に属しているか
  - NG例: 「ダメージ計算」と「UI表示更新」が同一コンポーネントに混在
  - OK例: 「ダメージ計算」と「HP減算」が同一コンポーネント（両方 Health ドメイン）
- **変数群** がタスクの実行に必要なものだけで構成されているか
- カテゴリとドメインの整合性

### S3: メッセージレベルの SRP

- 1つのメッセージが複数の無関係な処理をトリガーしていないか
- メッセージに紐づくアクションが同一の関心事に属しているか

## レビュー結果の出力形式

### 問題がある場合

各問題をカテゴリ・重要度と共に報告する。

```
## レビュー結果

### 問題一覧

| # | 重要度 | カテゴリ | ファイル | 問題 | 修正案 |
|---|-------|---------|--------|------|-------|
| 1 | ERROR | C2 | player.md | 送信メッセージ "Attack" の送信先 "Enemy" が受信側に未記載 | enemy.md の受信メッセージに追加 |
| 2 | WARN  | S1 | game-manager.md | 概要に「スコア管理とUI更新」の2責務が混在 | ScoreManager と UIUpdater に分離を検討 |
| 3 | INFO  | C4 | renderer.md | Unity向けレンダリング指示が未記載 | URP/HDRP パイプライン情報を追記 |

### サマリー
- ERROR: {n} 件（修正必須）
- WARN: {n} 件（修正推奨）
- INFO: {n} 件（参考情報）
```

### 重要度の定義

| 重要度 | 意味 | アクション |
|-------|------|----------|
| **ERROR** | 整合性の破綻。コード生成に支障をきたす | 修正必須。自動修正が可能なら修正する |
| **WARN** | 設計品質の問題。コード生成は可能だが品質に影響 | 修正推奨。判断はユーザーに委ねる |
| **INFO** | 改善提案。なくても問題ない | 参考情報として報告のみ |

## 自動修正

以下の問題は検出時に自動修正する：

| 問題 | 修正内容 |
|------|---------|
| メッセージの双方向不一致 | 欠落側のファイルに受信/送信メッセージを追記 |
| インタフェースファイル欠落 | `interfaces/` にファイルを生成 |
| 空セクション見出し欠落 | テンプレートに従い見出しと「なし」を追記 |
| 命名規則違反 | ファイル名を kebab-case にリネーム |

それ以外の問題（SRP違反、設計判断を伴うもの）は報告のみ行い、修正はユーザーに委ねる。
